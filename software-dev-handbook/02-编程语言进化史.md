# 第二部分：编程语言的进化史

> 每一门语言的诞生，都不是因为「有人想造一门语言」。  
> 而是因为**有人遇到了一个当时所有语言都解决不了的问题**。

---

## 2.1 序章：史前时代（1950s）

在 C 语言出现之前，编程世界已经有了一些先行者：

### FORTRAN（1957）—— 第一个高级语言

IBM 的工程师们面临一个问题：IBM 704 计算机卖给了很多科研机构，但每个机构都要从头用汇编写计算程序。数学家和物理学家不想学汇编。

**解决方案**：让代码长得像数学公式。

```fortran
C = A + B
IF (C .GT. 10) C = 0
```

FORTRAN 的编译器是有史以来第一个**优化编译器**——因为它面向的客户是科学家，他们关心计算结果的正确性，但更关心「能不能算得更快」。这个传统延续至今：FORTRAN 在数值计算领域的性能仍然是顶级的。

### COBOL（1959）—— 给商业世界的语言

美国国防部需要一种语言来处理大量的商业数据（工资单、库存、财务报表）。FORTRAN 太「数学化」了，不适合描述「如果客户等级是 VIP 则打 8 折」这类业务逻辑。

COBOL 的设计目标：**代码读起来像英文**。

```cobol
IF CUSTOMER-TYPE = "VIP"
   MULTIPLY PRICE BY 0.8 GIVING FINAL-PRICE
```

COBOL 至今仍在运行——全球约 70% 的 ATM 交易和大量银行的底层系统仍依赖 COBOL 代码。这不是因为它好，而是因为**重写的风险和成本太高**。

### LISP（1958）—— 一个异类

MIT 的 John McCarthy 想要研究「机器能否思考」。他发现 FORTRAN 不适合处理符号逻辑和递归。于是他造了 LISP——一门把**代码和数据当作同一种东西**的语言。

LISP 引入了太多超前的概念（垃圾回收、递归、高阶函数），在当时几乎没人理解。但这些概念后来被几乎每一门现代语言吸收。LISP 不是「失败了」，它是「来早了 30 年」。

## 2.2 C 语言（1972）：贝尔实验室的地下项目

### 工程危机：每个计算机都要重写一遍操作系统

1969 年，贝尔实验室的 Ken Thompson 和 Dennis Ritchie 有一个「业余项目」：写一个叫 UNIX 的操作系统。

最初他们用**汇编**写。问题来了：DEC 公司出了一款新计算机 PDP-11，他们想把 UNIX 移植过去。这意味着——全部用汇编重写一遍。几千行代码，逐行翻译，调试到崩溃。

Thompson 尝试用当时的一门语言 B 来重写，但 B 语言太简陋了——没有数据类型，性能差到无法用于操作系统。

### C 的诞生：一门「刚好够用」的语言

Ritchie 在 B 的基础上做了两个关键改进：

1. **加入了数据类型**（int、char、pointer 等）——直接对应内存中的存储方式
2. **保留了指针**——允许直接操作内存地址

这让 C 拥有了一种独特的**双重人格**：

```
高层的一面                 底层的一面
─────────────────────────────────────────
if (x > 10) { ... }      可以直接读写内存地址
结构化的控制流             可以操作寄存器
函数调用                   可以内联汇编
```

Ritchie 的哲学是：**不给程序员强加任何抽象。** C 假设你知道自己在做什么，并且允许你做任何事——包括把内存写坏。

1973 年，UNIX 内核用 C 重写完成。从此，**只要一个新平台上有 C 编译器，UNIX 就能移植过去**。这是软件史上的一个分水岭：操作系统的可移植性实现了。

### C 解决的核心问题

| 问题 | C 的答案 |
|:---|:---|
| 性能 | 编译成机器码，几乎无运行时开销 |
| 可移植性 | 一套代码 + 不同平台的编译器 → 到处运行 |
| 对硬件的控制力 | 指针 + 手动内存管理 → 精确到字节 |

C 如此成功，以至于它成了计算机世界的**通用语**。今天几乎每一个操作系统的内核、每一种语言的运行时，底层都是 C 写的。Windows 内核是 C，Linux 内核是 C，Python 的解释器 CPython 是 C，Java 的 JVM 是 C++（C 的后代）。

## 2.3 C++（1983）：当项目大到一个人管不过来

### 工程危机：软件规模爆炸

1979 年，丹麦程序员 Bjarne Stroustrup 在贝尔实验室工作。他在分析 UNIX 内核的代码时，发现了一个令人不安的趋势：

**项目越大，越难以用纯 C 管理。**

举个例子：你写了一个处理「文件」的模块。后来需求变了，要支持网络流（socket），本质上也是「读数据、写数据」，但文件是 `FILE*`，网络是 `int fd`——**类型不同，代码得重写。**

更糟的是，在一个 10 万行的 C 项目里：
- 你怎么保证一个函数不会不小心修改别人的数据？
- 你怎么知道一个 `void*` 指针到底指向什么类型？
- 你调用了一个函数，它释放了你还在用的内存——怎么避免？

这些问题的本质是：**C 没有提供任何机制来组织大型程序的逻辑结构。**

### C++ 的答案：把现实世界的组织方式映射到代码

Stroustrup 引入了 **面向对象编程（OOP）**。核心思想：**把数据和处理数据的函数打包在一起，形成一个「对象」**。

```cpp
// 纯 C 的写法：数据和操作是分离的
struct File {
    char* path;
    int fd;
};
void file_open(File* f, const char* path);
void file_read(File* f, char* buf);

// C++ 的写法：数据和方法封装在一起
class File {
    char* path;
    int fd;
public:
    void open(const char* path);
    void read(char* buf);
};
```

表面上看只是写法不同。但在一个 50 万行的项目里，这个区别是**生死之别**：

- **封装**：对象的内部数据不能被外部随意修改 → 减少了意外错误
- **继承**：`NetworkStream` 可以继承自 `Stream`，复用已有的读写逻辑 → 不重写代码
- **多态**：同一个接口 `read()`，不同的对象有不同的实现 → 代码更灵活

### C++ 的哲学：「零开销抽象」

Stroustrup 定了一条铁律：**如果你不用一个特性，你就不为它付出任何性能代价。**

这导致 C++ 是一门极复杂的语言——它几乎包含了 C 的全部功能，加上面向对象，后来又加上泛型（模板）、函数式特性、自动内存管理（智能指针）。到今天，C++ 标准超过 2000 页。

但它的定位一直没变：**当你需要极致的性能（像 C 一样快），同时又需要管理大型项目的复杂度时，C++ 是唯一的选择。** 游戏引擎（Unreal）、浏览器（Chrome）、高频交易系统、数据库（MySQL）——这些领域的核心都是 C++。

## 2.4 Java（1995）：一次编写，到处运行

### 工程危机：软件的「安装」地狱

1991 年，Sun Microsystems 的 James Gosling 和他的团队在做一个叫「Green」的项目——给家用电器写嵌入式程序。

他们最初用 C++。但家电的 CPU 五花八门（ARM、MIPS、SPARC……），每换一种 CPU 就要重新编译，还要针对不同平台调整代码。更糟的是，C++ 的指针和手动内存管理在嵌入式设备上导致了**大量难以调试的 bug**——一个空指针就可能让微波炉死机。

Gosling 开始设计一种新语言。他的目标非常明确：

> **写一次代码，在任何设备上都能跑。**

这个想法在当时是相当激进的。实现它的方式是两层抽象：

```
Java 源代码
    ↓ 编译（javac）
Java 字节码（.class 文件）
    ↓ 运行在 JVM（Java 虚拟机）上
实际的 CPU
```

关键创新是 **JVM（Java Virtual Machine）**——一个**软件模拟的计算机**。JVM 为每种操作系统和 CPU 都提供了一个实现，但它对上层 Java 程序暴露的接口是完全一样的。

### Java 的设计哲学

除了跨平台，Java 还做了几个大胆的决定：

**1. 干掉指针**

程序员不能直接操作内存地址。这消除了 C/C++ 中最危险的一类 bug：野指针、缓冲区溢出。

**2. 自动垃圾回收（Garbage Collection）**

C/C++ 中 `malloc` / `free` 必须手动配对，忘了 `free` 就内存泄漏，`free` 早了就用到了已释放的内存。Java 把内存回收交给 JVM 自动处理。

**3. 一切都是对象**

不像 C++ 还保留了原始类型和全局函数，Java 强制一切代码都在类里。

### Java 为什么赢了？

1995 年，Netscape 浏览器宣布支持 Java Applet——在网页里运行 Java 程序。这是革命性的。随后几年：

- 企业级开发（服务器后端）：Java 的稳定性、跨平台性和丰富的类库让它成为银行、保险、电商的标配
- Android（2008）：Google 选择 Java 作为 Android 的主要开发语言（虽然是运行在自己的 Dalvik/ART 虚拟机上）
- 大数据生态：Hadoop、Spark、Kafka、Elasticsearch——这些基础设施几乎都是 Java/Scala（JVM 语言）

Java 的 slogan 从「一次编写到处运行」悄悄变成了 **「招到的人都会写」**。它最大的资产不是技术特性，而是**全球最大的开发者社区和类库生态**。

## 2.5 Python（1991）：不是最好的语言，但可能是最合适的

### 工程危机：编程的门槛太高了

1991 年，荷兰程序员 Guido van Rossum 在圣诞节假期做了一个项目。

他当时在工作中使用 C 写系统工具，用 Unix Shell 写脚本。Shell 脚本很方便但功能太弱，C 功能强大但写起来太慢。他想：

> 能不能有一种语言——像 Shell 一样随手就能写，但像 C 一样能做正经事？

这就是 Python 的起点。Guido 的目标不是「造一门完美的语言」，而是**让编程变得容易。** 他把这称为「计算机为人类服务」（Computer Programming for Everybody）。

### Python 的设计哲学

Python 有两个核心设计原则：

**1. 可读性优先**

强制用缩进表示代码块，而不是花括号。初看这是「审美」问题，其实是工程决策：任何一个 Python 程序员都能读懂另一个 Python 程序员的代码结构，减少了沟通成本。

**2. 「做一件事应该只有一种明显的方法」**

这和 Perl 的「There's more than one way to do it」正好相反。Python 社区有 PEP 8 编码规范，有严格的哲学（Python 之禅 import this），所有库都倾向于用统一风格。

### Python 为什么在 2010 年代爆发？

Python 诞生后默默无闻了将近 20 年。它被用来写运维脚本、教学编程、做简单的 Web 开发。真正让它炸裂的是三个浪潮：

1. **数据科学**（2010s）：NumPy、pandas、matplotlib 让 Python 成了科学计算的标配
2. **机器学习/AI**（2015s）：TensorFlow、PyTorch 选择了 Python 作为前端语言
3. **DevOps/云原生**：Ansible、Airflow、Jupyter Notebook

Python 的成功说明了软件工程中一个重要原则：**语言本身不需要最强，只要在关键生态位上做到「最顺手」，社区就会把它推到顶峰。** Python 慢吗？慢。但 AI 研究的瓶颈是算法设计，不是代码运行速度——真正的计算由底层的 C/C++/CUDA 完成。

## 2.6 JavaScript（1995）：10 天写出来的语言，意外统治了世界

### 工程危机：网页太无聊了

1995 年，Netscape 浏览器占据了 80% 以上的市场份额。但网页只能展示静态的文本和图片——没有动画，没有交互，没有表单验证。

Netscape 的管理层做了一个决定：我们需要一种**在浏览器里运行的脚本语言**。

他们找到了 Brendan Eich，给了他一个任务：**为浏览器设计一门语言，10 天内完成**。

Eich 在 10 天内完成了第一版。这门语言被命名为 JavaScript（主要是为了蹭当时热门的 Java 的名气——一个营销决定，至今仍在造成困惑）。

### JavaScript 的设计特点

因为只写了 10 天，JavaScript 有很多「粗糙」的设计：

- 没有整数类型，只有 64 位浮点数
- `==` 的类型转换极其诡异（`[] == ![]` 返回 `true`）
- `this` 的绑定规则让初学者崩溃
- 所有东西都是可变的（mutable）

但 Eich 在匆忙中做对了**三件关键的事**：

**1. 函数是一等公民**

JS 中函数可以赋值给变量、作为参数传递、在运行时创建。这让它天生支持**回调（callback）**和**闭包（closure）**，后来成为异步编程的基础。

**2. 原型继承**

不是 Java/C++ 的类继承，而是「对象直接继承对象」。虽然很多人觉得不好理解，但它非常灵活。

**3. 事件驱动模型**

浏览器里的一切都是事件：点击、滚动、网络响应。JS 天然适合这种「等着事情发生然后响应」的编程模式。

### JavaScript 如何从玩具变成王者

| 年份 | 事件 | 影响 |
|:---|:---|:---|
| 2004 | Google 发布 Gmail | 证明 JS 可以做复杂的 Web 应用 |
| 2006 | jQuery 出现 | 解决了浏览器兼容性问题，JS 开发效率暴增 |
| 2008 | V8 引擎（Chrome） | 即时编译（JIT）让 JS 性能提升了 10-100 倍 |
| 2009 | Node.js 发布 | JS 跑到了服务器端——一门语言统治前后端 |
| 2013 | React 发布 | 组件化思维彻底改变了前端开发方式 |
| 2015 | ES6（ECMAScript 2015）| 加入了 class、箭头函数、模块系统，JS 终于「成年」|

现在，JavaScript 是世界上使用人数最多的编程语言（没有之一）。地球上每一台有浏览器的设备都在运行它。

### TypeScript（2012）：给 JavaScript 装上安全带

微软的 Anders Hejlsberg（他也是 C# 和 Turbo Pascal 的设计者）面对一个问题：JavaScript 的项目越来越大——几千行、几万行、几十万行——但没有类型系统，重构代码就像在黑暗中拆炸弹。

TypeScript 是 JavaScript 的**超集**：所有合法的 JS 代码都是合法的 TS 代码。但 TS 增加了：

- **静态类型检查**：编译时就能发现「把字符串传给了需要数字的函数」这类错误
- **更好的 IDE 支持**：自动补全、跳转定义、重构——JS 做不到精确的这些操作
- **接口（Interface）**：定义数据结构的「合同」

TypeScript 的成功不是因为它「比 JS 强」，而是因为它**解决了 JS 在大型项目中最大的痛点。**

## 2.7 Rust（2015）：Mozilla 给出的内存安全答案

### 工程危机：70% 的安全漏洞都来自同一个问题

Mozilla 的工程师在维护 Firefox（主要是 C++ 写的）时，发现了一个惊人的统计：**所有被报告的安全漏洞中，约 70% 都与「内存管理错误」有关。**

- 缓冲区溢出：往一个 10 字节的数组里塞了 20 字节 → 覆盖了相邻内存 → 可能被黑客利用
- 使用已释放的内存（use-after-free）→ 可能读到垃圾数据，也可能被注入恶意代码
- 数据竞争：两个线程同时修改同一块内存 → 结果取决于谁先到

C/C++ 给了程序员全部的控制权，但**没有工具帮助检查这些错误**。几十年来，这些错误造成的经济损失可能超过几十亿美元。

### Rust 的激进方案

Rust 的创始人 Graydon Hoare 提出了一个全新的思路：

> 能不能在**编译期**就发现所有内存错误，而不是等到运行时崩溃？

这是 Rust 最核心的创新：**所有权（Ownership）系统**。

```
每个值在任何时刻，有且只有一个「所有者」。
当所有者离开作用域，值被自动释放。
引用有两种：
  - 不可变引用（&）：可以同时存在多个
  - 可变引用（&mut）：同一时刻只能存在一个
不可变引用和可变引用不能同时存在。
```

这听起来像是一堆规则，但效果是惊人的：

- **不需要垃圾回收器**（Java/Python/Go 都需要）→ 运行时零开销
- **不需要手动 free**（C/C++ 需要）→ 编译器自动插入正确的释放代码
- **不需要担心数据竞争**→ 编译器在编译期就能拒绝不安全的并发代码
- **不需要担心空指针**→ Rust 没有 null，用 Option 类型显式处理「可能有值也可能没有」

### Rust 的定位

Rust 不是为了取代 C/C++，而是**在需要极致性能和极致安全同时满足的场景下，提供唯一可行的选择。**

- Mozilla 用 Rust 重写了 Firefox 的 CSS 引擎（Servo），性能提升数倍
- Linux 内核从 2022 年开始接受 Rust 代码
- Android 用 Rust 写系统组件，显著减少了内存相关的安全漏洞
- Microsoft 在 Windows 内核中用 Rust 重写部分组件

## 2.8 插曲：构建工具——为什么我们需要 Gradle、Maven、npm？

在讲完语言之后，你必须理解一个配套生态的重要性——**构建工具**。

### 问题：写一个小程序和写一个项目，是完全不同的事

写一个 `hello.c`，编译只需要一行：

```bash
gcc hello.c -o hello
```

但一个真实的 Java 项目（比如一个电商后端）是什么样的？

```
src/
├── main/java/com/example/
│   ├── controller/   # 接收 HTTP 请求
│   ├── service/      # 业务逻辑
│   ├── dao/          # 数据库访问
│   └── model/        # 数据模型
├── test/java/com/example/
│   └── service/      # 单元测试
└── resources/        # 配置文件
```

它依赖了 30 个第三方库，每个库还有自己的版本号和传递依赖。部署时要打包成单独的 JAR 文件。

**你不可能手动管理这些。** 这就是构建工具存在的意义：

| 语言/生态 | 构建工具 | 职责 |
|:---|:---|:---|
| Java | **Maven / Gradle** | 依赖管理、编译、测试、打包、发布 |
| Python | **pip + setuptools / poetry** | 依赖管理、虚拟环境、打包 |
| JavaScript | **npm / yarn / pnpm** | 依赖管理、脚本运行、打包 |
| C/C++ | **CMake / Make** | 编译控制、依赖管理（较弱） |
| Rust | **Cargo** | 依赖管理、编译、测试、发布、文档——全包 |

构建工具的三大职责：

1. **依赖管理**：声明你需要什么库、什么版本，工具自动下载
2. **构建自动化**：编译 → 测试 → 打包，一键完成
3. **标准化**：团队里所有人用同样的方式构建项目，不会出现「在我机器上能跑」

**Gradle** 是 Java 生态中的一个里程碑。相比它的前辈 Maven（XML 配置），Gradle 用 Groovy/Kotlin 脚本定义构建过程，既灵活又强大。Android 的官方构建工具就是 Gradle。

用一张表来总结语言的演化逻辑：

| 语言 | 诞生年代 | 试图解决的核心问题 | 核心创新 |
|:---|:---|:---|:---|
| C | 1972 | 操作系统不可移植 | 贴近硬件的抽象 + 跨平台编译器 |
| C++ | 1983 | 大项目难以管理 | 面向对象（封装、继承、多态）|
| Java | 1995 | 跨平台 + 内存不安全 | JVM 虚拟机 + 自动垃圾回收 |
| Python | 1991 | 编程门槛太高 | 可读性优先 + 动态类型 |
| JavaScript | 1995 | 网页需要交互 | 浏览器内置 + 事件驱动 + 函数一等公民 |
| TypeScript | 2012 | JS 大项目失控 | 静态类型超集 |
| Rust | 2015 | 70% 的安全漏洞来自 C/C++ 内存错误 | 编译期所有权检查 |

**一条主线**：每一代语言都在试图把**人类容易犯的错误**交给**机器在编译期自动检查**，同时保持或提升对底层硬件的表达能力。
