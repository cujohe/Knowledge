## 4.1 语言的血缘关系

```
          ALGOL (1958)
           │
           ├────────────── C (1972)
           │                │
           │                ├── C++ (1983) ────────── Java (1995)
           │                │    │                      │
           │                │    └── Rust (2015)         ├── C# (2000)
           │                │    (借鉴 C++ 的 RAII       ├── Scala (2004)

           │                │

           │
           ├── Smalltalk (1972) ── 启发了 Java 和 Python 的 OOP 设计
           │
           ├── Self (1987) ────── JavaScript (1995) 的原型继承
           │                         │

                                    (lambda、闭包、高阶函数)
```

## 4.2 同一个「Hello World」在不同语言中的样貌

```c
// C
#include <stdio.h>
int main() {
    printf("Hello, World!\n");
    return 0;
}
```

```cpp
// C++
#include <iostream>
int main() {
    std::cout << "Hello, World!" << std::endl;
    return 0;
}
```

```java
// Java
public class Main {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}
```

```python
# Python
print("Hello, World!")
```

```javascript
// JavaScript (Node.js)
console.log("Hello, World!");
```

```rust
// Rust
fn main() {
    println!("Hello, World!");
}
```

这六种写法在代码量上的差异，恰好反映了它们在抽象阶梯上的位置。Python/JS 一行搞定——因为解释器帮你处理了所有底层细节。C 需要 `#include` 和 `main` 函数——因为编译器需要你明确声明一切。

## 4.3 从同一段业务逻辑看各语言的思维差异

假设需求是：**读取用户列表，筛选出成年人，按姓名排序，打印。**

### Python（声明式、高抽象）

```python
users = load_users()                              # 假设返回 list[dict]
adults = [u for u in users if u["age"] >= 18]     # 一行筛选
adults.sort(key=lambda u: u["name"])              # 一行排序
for u in adults:
    print(u["name"])
```

### JavaScript（函数式、链式）

```javascript
const adults = users
    .filter(u => u.age >= 18)
    .sort((a, b) => a.name.localeCompare(b.name));

adults.forEach(u => console.log(u.name));
```

### Java（面向对象、类型安全）

```java
List<User> adults = users.stream()
    .filter(u -> u.getAge() >= 18)
    .sorted(Comparator.comparing(User::getName))
    .toList();

adults.forEach(u -> System.out.println(u.getName()));

// 但你需要先定义 User 类
public class User {
    private String name;
    private int age;
    // getter / setter / constructor...
}
```

### C++（控制力、零开销）

```cpp
std::vector<User> adults;
std::copy_if(users.begin(), users.end(), std::back_inserter(adults),
             [](const User& u) { return u.age >= 18; });

std::sort(adults.begin(), adults.end(),
          [](const User& a, const User& b) { return a.name < b.name; });

for (const auto& u : adults) {
    std::cout << u.name << '\n';
}
```

### Rust（安全 + 函数式）

```rust
let mut adults: Vec<&User> = users.iter()
    .filter(|u| u.age >= 18)
    .collect();

adults.sort_by(|a, b| a.name.cmp(&b.name));

for u in &adults {
    println!("{}", u.name);
}
```

### C（最底层，完全手动）

```c
// 筛选
User* adults = malloc(MAX_USERS * sizeof(User));
int adult_count = 0;
for (int i = 0; i < user_count; i++) {
    if (users[i].age >= 18) {
        adults[adult_count++] = users[i];
    }
}

// 排序（需要自己写或调用 qsort）
int compare_by_name(const void* a, const void* b) {
    return strcmp(((User*)a)->name, ((User*)b)->name);
}
qsort(adults, adult_count, sizeof(User), compare_by_name);

// 打印
for (int i = 0; i < adult_count; i++) {
    printf("%s\n", adults[i].name);
}
free(adults);  // 别忘了！
```

同一段逻辑：
- Python/JS：几行代码，心智负担几乎为零
- Java：类型声明带来了额外的代码量，但 IDE 支持最好
- C++：与 Java 类似，但更灵活（也更危险）
- Rust：最接近函数式风格，同时保证了安全
- C：你需要手动管理一切——每一步都是对硬件的精确控制

## 4.4 语言选择决策矩阵

当你真正开始一个项目时，怎么选语言？以下矩阵基于实际工程经验：

| 场景 | 首选 | 次选 | 不要选 |
|:---|:---|:---|:---|
| **Web 后端（业务密集）** | Java + Spring | Go / C# | C / C++ |
| **Web 后端（快速原型）** | Python + FastAPI | Node.js | C / Rust |
| **Web 前端** | TypeScript + React | JavaScript + Vue | 其他语言编译到 WASM |
| **移动 App（原生）** | Kotlin / Swift | Flutter (Dart) | C++ (除非是游戏) |
| **AI / 机器学习** | Python | C++ (底层引擎) | Java / JS |
| **数据科学** | Python | R | C |
| **嵌入式 / IoT** | C | Rust | Python |
| **操作系统内核** | C / Rust | C++ | Java / Python |
| **游戏引擎** | C++ | Rust | Python / JS |
| **CLI 工具** | Rust / Go | Python | Java（启动太慢）|
| **高频交易** | C++ | Rust | Python |

## 4.5 一个真实项目中各语言如何协作

假设你在做一个典型的互联网产品——比如一个电商 App：

```
┌──────────────────────────────────────────┐
│          前端（用户的手机/浏览器）           │
│                                           │
│  React (TypeScript)  →  用户界面           │
│  调用后端 REST API                        │
└──────────────────┬───────────────────────┘
                   │ HTTP/JSON
┌──────────────────▼───────────────────────┐
│          API 网关层（后端）                 │
│                                           │
│  Nginx (C)  →  反向代理 + 负载均衡          │
│  转发请求到 Java 服务                       │
└──────────────────┬───────────────────────┘
                   │
┌──────────────────▼───────────────────────┐
│         业务服务层（Java/Spring）            │
│                                           │
│  用户服务  →  订单服务  →  支付服务          │
│  处理业务逻辑 + 事务管理                     │
│  用 Gradle 构建管理                         │
└──────────────────┬───────────────────────┘
                   │
┌──────────────────▼───────────────────────┐
│         数据处理层（Python）                 │
│                                           │
│  推荐系统 (PyTorch, C++ 底层)               │
│  数据分析 (pandas, NumPy, C 底层)           │
│  定时任务 (Airflow)                         │
└──────────────────┬───────────────────────┘
                   │
┌──────────────────▼───────────────────────┐
│         基础设施                             │
│                                           │
```

注意：在这个项目中，你看到的语言有 TypeScript、Java、Python、C、C++、Go。它们不是竞争关系——它们**各司其职**。

## 4.6 抽象阶梯：一张终极地图

把六门语言放在第一部分建立的「抽象阶梯」上：

```
                    抽象程度
                       ↑
                       │
    Python ────────   高   │  「我关心要做什么，不关心怎么做」
    JavaScript            │  自动内存管理，动态类型
    TypeScript            │
                       │
    Java ───────────   中   │  「我关心对象关系，但不管内存」
    C#                   │  自动内存管理，静态类型
                       │
    C++ ───────────   中低  │  「我控制了内存，但用了抽象」
    Rust                  │  手动/编译期内存管理，丰富抽象
                       │
    C ─────────────   低   │  「我控制了一切」
    （汇编）              │  手动内存管理，几乎没有抽象
                       │
    硬件 ──────────   无   │  晶体管、逻辑门、CPU
                       │
                       └──────────────────→ 控制力
```

**关键洞察**：抽象程度越高，开发效率越高；控制力越强，性能天花板越高。选择哪门语言，本质上是选择你愿意为「开发效率」付出多少「性能/控制力」的代价。

## 4.7 构建工具的演化地图

我们已经看到了各种构建工具。它们之间的逻辑关系：

```
Make (1976)  ——  最早的构建工具，基于文件时间戳
    │
    ├── CMake (2000)  ——  Make 的跨平台替代品（C/C++ 主流）
    │
    ├── Maven (2004)  ——  Java 世界的第一代标准化构建工具（XML 配置）
    │       │
    │       └── Gradle (2012)  ——  Maven 的现代替代品（Groovy/Kotlin 脚本）
    │
    ├── npm (2010)  ——  Node.js 的包管理器
    │       │
    │       ├── yarn (2016)  ——  npm 的更快替代
    │       └── pnpm (2017)  ——  更高效的磁盘使用
    │
    ├── pip (2011)  ——  Python 的包管理器
    │       │
    │       ├── pipenv (2017)  ——  虚拟环境 + 依赖锁定
    │       ├── poetry (2018)  ——  现代 Python 包管理（像 Cargo）
    │       └── uv (2024)  ——  Rust 写的超快 pip 替代
    │
    └── Cargo (2015)  ——  Rust 的构建工具（被公认为最佳设计）
```

**共同趋势**：更快的解析速度、更可靠的依赖锁定、更一体化的体验（编译+测试+发布+文档）。Cargo 代表了这条演化路径的当前终点。
