# 01_需求分析与项目规划：将大需求拆解成可执行的步骤

在本章中，我们将学习如何将一个宏观的“大需求”分解为具体、可操作的小功能，并规划项目的整体结构。这是任何生产实践项目的第一步，也是至关重要的一步。

## 1. 什么是需求分析？为什么重要？

想象一下，你的客户（或者你自己）说：“我需要一个任务管理系统。” 这就是所谓的“大需求”。
如果直接开始写代码，你可能会遇到以下问题：
*   不知道从何开始。
*   写了一部分后发现逻辑不通。
*   功能实现后不符合最初的设想。
*   代码结构混乱，难以维护和扩展。

**需求分析 (Requirements Analysis)** 的目的就是将这种模糊的愿望转化为清晰、明确、可实现的功能列表。它帮助我们理解：
*   **用户是谁？** (谁会用这个系统？)
*   **用户想做什么？** (他们期望系统提供哪些功能？)
*   **系统应该如何响应？** (系统在接收到输入后应该有什么行为？)
*   **有哪些限制条件？** (性能要求、安全要求、技术栈限制等)

通过需求分析，我们可以形成一个共同的理解，避免返工，提高开发效率和产品质量。

## 2. 案例需求：任务管理系统

我们的“大需求”是：“开发一个简单的任务管理 RESTful API”。
现在，让我们来拆解它。一个任务管理系统应该具备哪些最基本的功能呢？

### 用户故事 (User Stories)

我们可以从用户的角度来描述功能，这有助于更好地理解需求：

*   **作为一个用户，我希望能创建新的任务**，包括标题、描述、状态和截止日期，以便记录我待办的事情。
*   **作为一个用户，我希望能查看所有任务的列表**，以便概览我的所有待办事项。
*   **作为一个用户，我希望能查看单个任务的详细信息**，以便了解任务的具体内容。
*   **作为一个用户，我希望能更新任务的状态、描述或截止日期**，以便修改任务进度或详情。
*   **作为一个用户，我希望能删除已完成或不需要的任务**，以便清理我的任务列表。

从这些用户故事中，我们可以提炼出具体的 API 操作。

### API 功能与端点设计

由于我们要做的是一个 **RESTful API**，我们需要思考“资源”是什么，以及如何通过 HTTP 方法对其进行操作。

*   **资源 (Resource)**：在我们的案例中，核心资源是 `Task` (任务)。
*   **HTTP 方法 (HTTP Methods)**：
    *   `GET`：获取资源
    *   `POST`：创建资源
    *   `PUT`/`PATCH`：更新资源
    *   `DELETE`：删除资源

根据用户故事和 RESTful 原则，我们可以设计出如下的 API 端点：

| 功能           | HTTP 方法 | 端点 (URL)              | 请求体 (Request Body)                   | 响应体 (Response Body)                                 |
| :------------- | :-------- | :---------------------- | :-------------------------------------- | :----------------------------------------------------- |
| 创建任务       | `POST`    | `/tasks`                | `{"title": "...", "description": "...", "status": "...", "due_date": "..."}` | 新创建的任务对象，包含 ID                              |
| 获取所有任务   | `GET`     | `/tasks`                | 无                                      | 任务列表 `[{}, {}, ...]`                               |
| 获取单个任务   | `GET`     | `/tasks/<task_id>`      | 无                                      | 单个任务对象 `{}` 或 404 Not Found                    |
| 更新任务       | `PUT`     | `/tasks/<task_id>`      | `{"title": "...", "status": "..."}` (部分或全部字段) | 更新后的任务对象 `{}` 或 404 Not Found                |
| 删除任务       | `DELETE`  | `/tasks/<task_id>`      | 无                                      | 空响应 (204 No Content) 或 404 Not Found             |

**任务属性 (Task Attributes)**:
*   `id`: 任务的唯一标识符 (整数，系统自动生成)
*   `title`: 任务标题 (字符串，必填)
*   `description`: 任务描述 (字符串，可选)
*   `status`: 任务状态 (字符串，如 "pending", "in_progress", "completed")
*   `due_date`: 截止日期 (字符串，例如 "YYYY-MM-DD" 格式，可选)
*   `created_at`: 创建时间 (时间戳，系统自动生成)
*   `updated_at`: 更新时间 (时间戳，系统自动生成)

## 3. 技术选型 (Tech Stack)

对于本次教程，我们将选择以下技术栈：

*   **后端框架**: **Flask**
    *   **为什么选择 Flask?** Flask 是一个轻量级的 Python Web 框架，被称为“微框架”。它不强制使用特定的工具或库，给予开发者极大的自由度。这对于初学者理解 Web 开发的核心概念非常有帮助，也适合构建小型的 API 服务。
*   **数据库**: **SQLite**
    *   **为什么选择 SQLite?** SQLite 是一个嵌入式数据库，不需要独立的服务器进程。数据存储在一个文件中，非常适合开发、测试和小型应用。它简化了数据库的配置和管理，让我们能更专注于应用逻辑。
*   **ORM (Object-Relational Mapper) 工具**: **SQLAlchemy (配合 Flask-SQLAlchemy)**
    *   **为什么选择 SQLAlchemy?** SQLAlchemy 是 Python 中功能强大且灵活的 ORM 工具，可以将 Python 对象映射到数据库表。`Flask-SQLAlchemy` 是 Flask 的一个扩展，它封装了 SQLAlchemy，使其在 Flask 应用中集成更加便捷。通过 ORM，我们可以用 Python 代码而不是纯 SQL 来操作数据库。
*   **依赖管理**: **pip** 和 **venv (Python 虚拟环境)**
    *   **为什么选择它们?** `pip` 是 Python 的包管理器，`venv` 是 Python 标准库中用于创建轻量级虚拟环境的模块。它们是 Python 项目中最常用和推荐的依赖管理方式，能有效隔离不同项目所需的库版本。
*   **版本控制**: **Git** 和 **GitHub**
    *   **为什么选择它们?** Git 是目前最流行的分布式版本控制系统，GitHub 是一个基于 Git 的代码托管平台。它们是现代软件开发中不可或缺的工具，用于追踪代码变更、协作开发和项目管理。
*   **测试框架**: **unittest (Python 内置)**
    *   **为什么选择 unittest?** `unittest` 是 Python 内置的测试框架，无需额外安装。它提供了编写单元测试和集成测试所需的基本工具。

## 4. 项目结构设计

一个清晰的项目结构是代码可维护性的基石。我们将规划 `code/task_app/` 目录下的文件和目录，使其职责明确。

```
code/
└── task_app/
    ├── venv/                       # Python 虚拟环境 (会被 .gitignore 忽略)
    ├── app.py                      # Flask 应用的入口文件，负责初始化应用和注册蓝图
    ├── config.py                   # 应用程序配置，如数据库连接字符串
    ├── models.py                   # 数据库模型定义 (SQLAlchemy)
    ├── routes.py                   # 定义 API 路由和视图函数
    ├── requirements.txt            # 项目依赖列表
    ├── .gitignore                  # Git 忽略文件配置
    └── tests/                      # 存放所有测试文件
        └── test_app.py             # API 的单元测试和集成测试
```

**文件职责说明：**

*   `app.py`:
    *   创建 Flask 应用实例。
    *   从 `config.py` 加载配置。
    *   初始化 `Flask-SQLAlchemy`。
    *   注册蓝图 (Blueprints)，即从 `routes.py` 导入 API 路由。
    *   启动 Flask 开发服务器。
*   `config.py`:
    *   定义 `Config` 类，存放应用的所有配置参数。
    *   例如：`SQLALCHEMY_DATABASE_URI`。
*   `models.py`:
    *   定义 `Task` 数据库模型。
    *   包含任务的各个属性及其数据类型。
*   `routes.py`:
    *   使用 Flask 蓝图 (Blueprint) 组织 API 路由。
    *   定义 `/tasks` 和 `/tasks/<task_id>` 的 GET, POST, PUT, DELETE 视图函数。
    *   处理请求数据，调用模型进行数据库操作，并返回 JSON 响应。
*   `requirements.txt`:
    *   列出项目所需的所有 Python 库及其版本。
*   `.gitignore`:
    *   指定 Git 应该忽略的文件和目录，例如 `venv/`、`__pycache__/` 和 SQLite 数据库文件 (`instance/test.db`)。
*   `tests/test_app.py`:
    *   包含对 `routes.py` 中定义的 API 端点的测试用例。
    *   使用 Flask 的测试客户端模拟 HTTP 请求。

通过这样的结构，我们可以确保代码的模块化，每个文件都负责特定的功能，方便开发、测试和未来的扩展。
