# 02_环境搭建与版本控制：打好项目开发的基础

在本章中，我们将学习如何为 Python 项目搭建一个干净、隔离的开发环境，以及如何使用 Git 进行版本控制。这两者都是现代软件开发中不可或缺的基础技能，能确保你的项目稳定、可协作。

## 1. Python 虚拟环境 (Virtual Environment)

### 什么是虚拟环境？

想象一下你同时在开发两个 Python 项目：项目 A 依赖 `Django 2.x`，而项目 B 依赖 `Django 3.x`。如果它们都直接安装在你的系统全局 Python 环境中，就会产生冲突，导致其中一个或两个项目无法正常运行。

**Python 虚拟环境** 就是为了解决这个问题而生。它是一个独立的、隔离的 Python 运行环境。每个虚拟环境都可以安装自己独立的 Python 包，而不会影响系统全局的 Python 环境，也不会影响其他虚拟环境。

**通俗比喻**：虚拟环境就像是一个独立的房间，你可以在里面随意布置家具（安装不同的库和版本），而不会影响到其他房间。

### 为什么使用虚拟环境？

*   **隔离性**：避免不同项目间的依赖冲突。
*   **清洁性**：保持系统全局 Python 环境的整洁，避免垃圾包。
*   **可移植性**：方便与团队成员共享项目依赖，确保 everyone's machine works the same。
*   **轻量化**：创建和删除都非常方便。

### 如何使用 `venv` (Python 3.3+ 内置)

Python 3.3 以后，标准库中内置了 `venv` 模块，我们无需安装额外的工具。

1.  **进入项目根目录**：
    首先，切换到你的项目目录。对于我们的任务管理 API，就是 `Knowledge/python-production-tutorial/code/task_app`。
    ```bash
    cd Knowledge/python-production-tutorial/code/task_app
    ```

2.  **创建虚拟环境**：
    在项目目录下运行以下命令来创建一个名为 `venv` 的虚拟环境。你可以给虚拟环境起任何名字，但 `venv` 是一个常见的约定。
    ```bash
    python -m venv venv
    ```
    这会在你的项目目录下创建一个 `venv/` 文件夹，里面包含了独立的 Python 解释器和 `pip`。

3.  **激活虚拟环境**：
    创建虚拟环境后，你需要“激活”它，这样你后续运行的 `python` 和 `pip` 命令都会指向这个虚拟环境而不是系统全局的。
    *   **macOS / Linux**:
        ```bash
        source venv/bin/activate
        ```
    *   **Windows (Command Prompt)**:
        ```cmd
        venv\Scripts\activate.bat
        ```
    *   **Windows (PowerShell)**:
        ```powershell
        venv\Scripts\Activate.ps1
        ```
    激活后，你的命令行提示符通常会显示 `(venv)` 字样，表示你当前正处于虚拟环境中。

4.  **安装项目依赖**：
    现在，你可以在激活的虚拟环境中安装项目所需的库了。我们的项目依赖都在 `requirements.txt` 文件中。
    ```bash
    pip install -r requirements.txt
    ```
    这会安装 Flask, Flask-SQLAlchemy 和 SQLAlchemy。这些库只会安装到当前的 `venv` 中。

5.  **退出虚拟环境**：
    当你完成项目开发或需要切换到其他项目时，可以退出当前虚拟环境。
    ```bash
    deactivate
    ```
    退出后，你的命令行提示符上的 `(venv)` 字样会消失，`python` 和 `pip` 命令会再次指向系统全局环境。

## 2. 版本控制与 Git / GitHub

### 什么是版本控制？

**版本控制系统 (Version Control System, VCS)** 是一种记录文件（特别是代码文件）内容变化，以便将来查阅特定版本修订情况的系统。它能帮助我们：
*   **追踪历史**：记录每一次代码修改，谁改了什么，什么时候改的。
*   **回溯版本**：可以轻松回到之前的任何一个版本，解决问题或撤销错误。
*   **多人协作**：允许多个开发者在同一个项目上并行工作，并有效合并各自的修改。
*   **代码备份**：将代码安全地存储在远程仓库。

**通俗比喻**：版本控制就像一个“代码时光机”和“协作平台”，让你能随时回到过去，也能和别人一起穿越未来。

### Git 基础操作

**Git** 是一个分布式版本控制系统，意味着每个开发者本地都有一份完整的代码仓库历史。

1.  **检查 Git 是否安装**：
    在终端输入 `git --version`。如果已安装，会显示版本信息；否则，你需要根据你的操作系统安装 Git。

2.  **克隆远程仓库**：
    对于我们这个教程，你已经从 GitHub 克隆了 `Knowledge` 仓库：
    ```bash
    git clone https://github.com/cujohe/Knowledge.git
    cd Knowledge
    ```

3.  **查看当前状态**：
    经常使用 `git status` 命令，它会告诉你哪些文件被修改了，哪些是新文件，哪些文件已暂存，以及当前所在的分支。
    ```bash
    git status
    ```

4.  **添加文件到暂存区 (Stage Area)**：
    在你修改或创建文件后，需要告诉 Git 哪些文件是你准备提交的。这通过 `git add` 命令完成。
    ```bash
    git add .      # 添加所有修改和新建的文件
    git add <file> # 添加特定文件
    ```

5.  **提交更改 (Commit)**：
    暂存区的文件可以被“提交”到本地仓库，形成一个版本历史记录点。每次提交都应该附带一个清晰的提交信息，说明本次提交做了什么。
    ```bash
    git commit -m "feat: Add initial task management API structure"
    ```
    **提交信息约定 (Commit Message Convention)**：
    通常我们遵循一定的提交信息格式，例如 `type: description`。
    *   `feat`: 新功能 (features)
    *   `fix`: 修复 bug (bug fixes)
    *   `docs`: 文档变更 (documentation)
    *   `style`: 格式调整 (whitespace, formatting, missing semi colons, etc)
    *   `refactor`: 重构 (refactoring production code)
    *   `test`: 添加测试 (adding missing tests, refactoring tests)
    *   `chore`: 构建过程或辅助工具的变更 (build system, CI, etc)

6.  **查看提交历史**：
    ```bash
    git log
    ```

7.  **分支管理 (Branching)**：
    分支允许你在不影响主线代码的情况下，独立地开发新功能或修复 bug。`main` (或 `master`) 通常是稳定版本的分支。
    *   **查看分支**：
        ```bash
        git branch
        ```
    *   **创建新分支**：
        ```bash
        git branch <new-branch-name>
        ```
    *   **切换分支**：
        ```bash
        git checkout <branch-name>
        ```
    *   **创建并切换到新分支** (常用)：
        ```bash
        git checkout -b <new-branch-name>
        ```

8.  **推送到远程仓库**：
    当你完成本地提交后，需要将这些提交推送到 GitHub 等远程仓库，与团队成员共享。
    ```bash
    git push origin <your-branch-name>
    ```
    `origin` 是远程仓库的默认别名。

### GitHub 工作流简介

**GitHub** 提供了一个可视化的平台，让 Git 的协作更加高效。典型的 GitHub 工作流如下：

1.  **克隆仓库**：将远程仓库复制到本地。
2.  **创建新分支**：为新功能或 bug 修复创建一个新的分支。
3.  **开发并提交**：在本地分支上进行开发，并多次提交。
4.  **推送到远程分支**：将本地分支的修改推送到 GitHub 上的对应分支。
5.  **发起 Pull Request (PR)**：在 GitHub 上发起一个 Pull Request。PR 是请求将你的分支合并到主分支 (如 `main`) 的请求。它提供了一个代码审查的平台，其他开发者可以在这里查看你的代码、提出建议、讨论。
6.  **代码审查与合并**：团队成员审查代码，通过后将 PR 合并到主分支。

在这个教程中，我为你创建了 `python-production-tutorial` 分支，并将所有内容推送到这个分支。最后我会为你发起一个 Pull Request，演示这个流程。

## 3. `.gitignore` 文件

在项目根目录 (对于我们的案例是 `code/task_app/`) 下，通常会有一个名为 `.gitignore` 的文件。这个文件的作用是告诉 Git 哪些文件或目录应该被忽略，不应该被添加到版本控制中。

**为什么需要 `.gitignore`？**
*   **避免不必要的文件**：例如虚拟环境 (`venv/`)、编译生成的 `.pyc` 文件、IDE 配置文件 (`.idea/`)、日志文件、数据库文件 (`*.db`) 等，这些文件通常是本地环境特有的，或者可以由其他文件生成，不需要追踪。
*   **保持仓库清洁**：避免提交无关紧要的文件，减少仓库大小，提高 `git clone` 和 `git pull` 的速度。
*   **避免安全风险**：不小心提交敏感信息（如 API 密钥、配置文件中的密码）到公共仓库。

我们已在 `code/task_app/` 目录下创建了一个 `.gitignore` 文件，内容如下：

```gitignore
# Python
__pycache__/
*.pyc
*.pyd
*.pyo

# venv
venv/
.venv/

# Database
instance/
*.db

# IDE
.idea/
.vscode/

# OS
.DS_Store
thumbs.db
```

**说明：**
*   `__pycache__/`：Python 字节码缓存目录。
*   `*.pyc`, `*.pyd`, `*.pyo`：Python 编译生成的文件。
*   `venv/`, `.venv/`：虚拟环境目录。
*   `instance/`, `*.db`：Flask 应用的 `instance` 目录通常用于存放本地数据库文件 (如 `tasks.db`)，这些是不需要版本控制的。
*   `.idea/`, `.vscode/`：常见 IDE (如 PyCharm, VS Code) 的项目配置文件。
*   `.DS_Store`, `thumbs.db`：操作系统生成的文件。

通过 `.gitignore`，我们可以确保只有真正重要的代码文件被纳入版本控制。
